Completed Features

✓

biz_step URN Validation

Every event is validated against the GDST Table C1 URN map before persistence.

✓

Ship/Receive Constraint Enforcement

Source and destination lists are required on ship and receive events; blocked on other CTEs.

✓

EPC Role Validation

Transformation and aggregation CTEs enforce correct input/output/parent/child EPC roles.

✓

Full KDE Hydration on GET

GET /events/{id} returns the event with all linked KDE sub-tables fully populated.

✓

Multi-Filter Event Search

Up to 9 independent filters on GET /events — CTE, party, location, product, date range, pagination.

✓

Chain-of-Custody History

GET /traceable-objects/{id}/history returns all events in which a product lot appeared.

✓

Atomic Event Creation

All KDEs (catch areas, certifications, welfare policies, EPC list) are written in a single transaction.

✓

Swagger / OpenAPI Docs

Interactive documentation auto-generated from handler annotations, served at /swagger/.

✓

Auto-Migrating Database

Goose migrations are embedded in the binary and applied automatically on startup.

✓

Structured Logging & Middleware

slog-based structured logging; request/response and panic-recovery middleware.

✓

Disposition Auto-Set & Validation

Disposition is automatically set to the Table C1 URN for each CTE; explicit values conflicting with the spec are rejected.

✓

GDST / GS1 URI Format Validation

EPC URIs (LGTIN, SSCC, GDST URN), party identifiers (PGLN), and location identifiers (SGLN) are validated on creation against Section 3 formats.

✓

bizTransactionList for Authorizations

Events support EPCIS bizTransactionList as an alternative to certificationList for fishing, transshipment, and landing authorization documents (Tables W2f, Aq2f).

✓

On-Vessel Processing Vessel Linkage

GET /events/{id} auto-hydrates Table W2a vessel KDEs when the event read point is a vessel location — surfaces IMO number, flag state, registry URL inline.

✓

SSCC Logistics Unit Identification

Traceable objects carry an identifier_type field — auto-set to logistics_unit for SSCC URIs and product_lot for all others (Section 3.2).

✓

Required EPCIS Technical Fields

product_owner_id, information_provider_id, and read_point are enforced as required on every event per GDST Table W1b.

✓

Table C1 EPC Count Rules

Commingling and packaging require N inputs + 1 output; aggregation and disaggregation require 1 parent + N children; all transformation events require ≥1 input and ≥1 output.

Suggested Next Steps

  • 1
    Authentication & Authorization — API key or JWT middleware; per-party access scoping so information providers can only read/write their own events.
  • 2
    Deployment Configuration — Dockerfile and/or Kubernetes manifests; environment-specific configuration for staging and production; database provisioning automation.
  • 3
    EPCIS 2.0 / GS1 Export — Serialise stored events as standard EPCIS 2.0 XML or JSON-LD so data can be shared with external supply-chain platforms and comply with digital transfer requirements.
  • 4
    List Endpoints for Master Data — Paginated GET /parties, GET /vessels, GET /locations to support front-end UIs and reporting dashboards.
  • 5
    Integration / E2E Tests — Tests that exercise full request flows against a real database to complement the existing unit test layer.
  • 6
    FAO Catch Area Code Validation — Validate that area_code values in catch areas are legitimate FAO, EEZ, or RFMO codes against published reference lists.
  • 7
    Customer Integration Inbound Adapter — An API endpoint (or webhook receiver) that accepts events in a customer's native format and maps them to GDST 1.2 EPCIS structure automatically. Would include a field-mapping layer, CTE type inference, and a dry-run validation mode so customers can test their data before committing it.
  • 8
    Operational Alerting (Slack / Discord) — Push structured notifications for key events (event creation, validation failures, inbound integration errors) to a Slack or Discord webhook. Useful for supply-chain operations teams monitoring data quality in real time without needing to tail logs.
